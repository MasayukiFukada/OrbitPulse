# リポジトリガイドライン

## プロジェクト概要（Overview）

本プロジェクトの目的や背景は以下の通り
- 目的
    - 個人用のスクラム管理システムを開発する
    - 休憩時間も考慮する
    - 可能ならゲーミフィケーションも取り入れたい
- 背景
    - 仕事でスクラムを取り入れたら結構楽しく感じた。
- 利用範囲
    - 個人

## ゴール

- 一人でも時間を有意義に使いながら計画的に様々なタスクを終わらせることができること

## ゴールではないもの

- 技術スタックを変更する
- 既存処理を推測すること

## 機能や要素

- Google カレンダー参照
- MCP 対応
    - 優先度は最低
- ダッシュボード
    - 様々な記録を見られる画面
- プロダクト(プロジェクト)バックログ
    - ストーリーや「Why(なぜ)」を重視
    - 見積はポイント制
    - 子としてスプリントバックログを複数持つ
- ToDoバックログ
    - プロダクト(プロジェクト)とまでは呼べないけれどやらなくてはいけないToDoタスクを複数持つ
- ToDoタスク
    - 親スプリントあり(未設定時はまだ実施未定)
    - 時間としてポモドーロのカラム(見積/実績)あり
    - 締切り日設定あり
    - 優先度あり
- スプリント
    - 関連するプロダクトバックログを複数登録可能
    - スプリント1回分
    - 開始日と終了日( MAX 14 日間までを想定 )
    - プランニングで決めた計画や、レトロスペクティブの内容を記録することもできる
- キャパシティ
    - 日付と結び付きを持つ
    - ポモドーロの個数として時間を持つ
- スプリントバックログ
    - 子として複数のスプリントタスクを持つ
- タスク
    - 親にスプリントバックログを持つ
    - 見積/実績を持つ
    - ポモドーロ制( 1ポモドーロ=25分 + 5分休憩 の回数)
        - 最低限の単位をどうするか
        - 0.5 ポモドーロを定義するべきか
- バーンダウンチャート

## コーディング規約（Coding Style Guidelines）

- **アーキテクチャ**: クリーンアーキテクチャ（Domain, Application, Infrastructure, Presentation）を採用。
- **スタイル**:
    - 関数は抽象レベルを揃え、1つの関数は短く保つ。
    - フォーマットと静的解析を遵守する。
- **テスト**: 
    - ドメインロジック、ユースケースについてはテストコードを作成する。


## セキュリティ（Security considerations）

- サーバーポート、DBパス等の設定は環境変数（`.env`）で管理する。
- 開発中は .env も読み書きして良い
- リポジトリに機密情報（APIキー等）を含めない。
